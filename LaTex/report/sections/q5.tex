\section{Q5 — Raft Failure Test Cases}

Five failure scenarios are scripted in \texttt{client/q5\_failure\_tests.py}.
Each test stops or disrupts one or more containers, observes system behavior,
then restores normal operation.

% ── Test Case 1 ──────────────────────────────────────────
\subsection*{Test Case 1 — Leader Crash}

\textbf{Setup:} Identify the current leader from logs, then stop that container.

\textbf{Expected behavior:} The remaining 4 nodes stop receiving heartbeats.
After their election timeouts expire (1.5–3s), one node initiates an election
and a new leader is elected. Operations resume once a new leader is established.

\textbf{Commands:}
\begin{lstlisting}[language=bash]
docker stop node2-api-server-a   # (or whichever node is leader)
# observe re-election in remaining nodes
docker start node2-api-server-a
\end{lstlisting}

% ── Test Case 2 ──────────────────────────────────────────
\subsection*{Test Case 2 — Single Follower Crash}

\textbf{Setup:} Stop one follower while the system is operational.

\textbf{Expected behavior:} The cluster retains a majority (4 of 5 nodes).
Log replication continues; the leader commits operations with ACKs from the
remaining 4 followers. The stopped node re-syncs its log when restarted.

% ── Test Case 3 ──────────────────────────────────────────
\subsection*{Test Case 3 — Two Followers Crash (Majority Retained)}

\textbf{Setup:} Stop two follower nodes simultaneously.

\textbf{Expected behavior:} 3 of 5 nodes remain — exactly a majority.
The leader can still reach consensus and commit operations. This is the
minimum viable cluster for the 5-node configuration.

% ── Test Case 4 ──────────────────────────────────────────
\subsection*{Test Case 4 — Majority Lost (System Stalls)}

\textbf{Setup:} Stop three nodes, breaking the majority threshold.

\textbf{Expected behavior:} The remaining 2 nodes cannot form a majority.
If the leader is among the 2 survivors, it will not commit any new operations
(cannot reach 3 ACKs). If a candidate election fires, it also cannot win.
The system stalls until nodes are restored.

  \label{fig:q5_t4}

% ── Test Case 5 ──────────────────────────────────────────
\subsection*{Test Case 5 — 2PC with Participant Down}

\textbf{Setup:} Stop one API server, then trigger a \texttt{UpdateStock2PC}
transaction via \texttt{twopc\_client.py}.

\textbf{Expected behavior:} The coordinator sends \texttt{VoteRequest} to all
5 participants. The stopped node is unreachable; the coordinator treats an
unreachable node as an implicit \texttt{VoteAbort}. \texttt{GlobalAbort} is
sent to all reachable participants and the transaction is rolled back.
